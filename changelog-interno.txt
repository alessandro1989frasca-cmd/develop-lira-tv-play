v1.0.128 — Scheda Programma + resume episodi + fix New Architecture
====================================================================
Data: aprile 2026
versionCode: 196

BUG FIX (sessione corrente)
---------------------------
- app/program-category.tsx: fix critico — EpisodeRow ora riceve
  progress={resumePositions[item.id]}. Prima resumePositions era caricato
  ma mai passato alla riga: barra di avanzamento e badge "RIPRENDI"
  non apparivano mai. Ora:
  · Barra di avanzamento blu (#60A5FA) sul fondo del thumbnail (stessa
    logica di VideoCard: soglia 2%-92% per nascondere valori irrilevanti)
  · Badge "RIPRENDI" in blu accanto a data/durata quando pct > 2% e < 92%
  · Badge "NUOVO" rimane se non c'è progresso e l'episodio è recente (<36h)
- app.json: newArchEnabled: false → true. Expo Go SDK 53 usa sempre la New
  Architecture: tenere false causava comportamento diverso tra test e
  build EAS di produzione. Ora allineati.

NUOVA FUNZIONALITÀ
------------------
- app/program-category.tsx: completamente ridisegnato come "Scheda Programma"
  per i programmi della categoria 'programmi'. Stile ispirato a streaming app
  premium, con identità visiva propria (accenti dorati #F5C518, distinta da RAI Play).
  Componenti: hero fullscreen con thumbnail ultima puntata + LinearGradient,
  barra dorata + label "Programma", titolo show grande, bottone "Ultima puntata"
  con cerchio play dorato, bottoni Condividi (Share nativo) e Preferiti (toggle
  cuore), descrizione del programma, sezione "Puntate" con lista episodi
  numerata (thumbnail, titolo, data, durata, badge NUOVO se <36h).
- Stack.Screen con headerShown: false — nessun header nativo, back button custom.
- FlatList con ListHeaderComponent per performance ottimale su liste lunghe.
- Badge "NUOVO" automatico per episodi pubblicati nelle ultime 36 ore.
- Bottone "Condividi" usa Share nativo React Native.
- Bottone "Preferiti" toggle locale (cuore pieno/vuoto).

BACKEND
-------
- backend/hono.ts: nuovo endpoint GET /api/program-description?name=...
  Recupera la descrizione del programma dall'API pubblica WordPress di liratv.it
  (wp-json/wp/v2/categories?slug=...). Cache in-memory con TTL 24 ore per slug,
  stripping HTML automatico. Conversione nome → slug WP (lowercase, accenti
  rimossi, spazi → trattini). Max ~33 programmi unici → 33 call esterne
  mai ripetute nella stessa giornata di server uptime.
- Questa feature non richiede modifiche a Supabase.

NOTE
----
- "Edizioni TG" e altre categorie non-programmi: comportamento invariato
  (navigazione classica con VideoCard).
- Descrizione: non disponibile per tutti i programmi (dipende dalle categorie WP).
  Se vuota, la sezione viene omessa automaticamente.
- La descrizione viene recuperata direttamente dall'API WordPress di liratv.it
  (wp-json/wp/v2/categories) senza passare dal backend — funziona su device
  fisico, Expo Go e web. Nessun problema CORS su native. React Query fa da
  cache per 24 ore: massimo 1 chiamata per programma per sessione.

v1.0.124 — Supabase failover per sezione News
=============================================
Data: aprile 2026
versionCode: 202

NUOVA FUNZIONALITÀ
------------------
- dataService.ts: aggiunto failover Supabase per fetchNewsArticles().
  Flusso: WordPress RSS → (fallback) Supabase cached_news → (fallback) AsyncStorage.
  Dopo ogni fetch WordPress riuscita, gli articoli vengono sincronizzati
  in background su Supabase (max 50 articoli, non bloccante).
  Richiede tabella cached_news su Supabase (vedi SQL nel changelog).

SQL TABELLA (da eseguire su Supabase SQL Editor):
  CREATE TABLE cached_news (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    link TEXT,
    pub_date TIMESTAMPTZ NOT NULL,
    author TEXT,
    category TEXT,
    thumbnail TEXT,
    description TEXT,
    content TEXT,
    video_url TEXT,
    updated_at TIMESTAMPTZ DEFAULT NOW()
  );

  che aggiorna il palinsesto prima che AsyncStorage carichi le preferenze) potrebbe
  schedulare una seconda notifica per lo stesso programma senza cancellare la prima.
  In pratica quasi impossibile con React 18 (automatic batching), ma è un punto fragile.
  Fix: aggiungere `singleNotifs` alle dipendenze dell'effetto e usare una ref per
  evitare il loop infinito (es. scheduledRef che traccia gli ID già processati).
